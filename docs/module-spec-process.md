# 模块功能说明流程

## 1. 目的

每个新模块都必须先形成可执行的功能说明，再进入实现，避免只写页面、不定义接口、不留测试依据。

统一流程：

```text
功能说明 -> 接口文档 -> 测试用例 -> 实现 -> 验证 -> 文档更新
```

适用范围：

- 首页模块
- token 看板模块
- Agent 日报模块
- 文章与项目说明模块
- 经验地图模块
- 后续新增的任何公开功能模块

## 2. 文件命名

每个模块创建一个独立文档：

```text
docs/feature-<module-slug>.md
```

示例：

```text
docs/feature-token-usage-dashboard.md
docs/feature-agent-signals.md
docs/feature-writing.md
```

## 3. 必填章节

每个模块文档必须包含：

- 文档状态：模块名称、所属里程碑、当前状态、关联规划。
- 功能目标：面向谁、解决什么问题、证明什么能力。
- MVP 范围：第一版做什么、不做什么。
- 数据来源：输入数据、采集方式、人工维护方式。
- 隐私边界：公开字段、禁止字段、脱敏规则。
- 接口文档：文件接口、API 接口、组件接口或数据契约。
- UI 状态：正常、空状态、错误状态、过期状态。
- 测试用例：用 ID 标记，覆盖 schema、计算、隐私、UI 和错误处理。
- 验收标准：用户可见结果和工程质量标准。
- 实现任务清单：按可提交的任务拆分。
- 待确认问题：实现前必须确认的风险点。

## 4. 接口文档要求

接口文档必须说明：

- 接口名称或文件路径。
- 消费方和生产方。
- 字段结构和字段含义。
- 必填字段和可选字段。
- 版本兼容策略。
- 错误处理策略。
- 哪些字段允许公开，哪些字段禁止公开。

如果模块没有外部 API，也必须写清楚内部数据接口或组件 props。

## 5. 测试用例要求

测试用例必须至少覆盖：

- Schema 校验：必填字段、类型、版本号。
- 数据计算：聚合、排序、窗口统计、边界值。
- 隐私扫描：账号、邮箱、API key、本机路径、原始 prompt、内部项目名。
- UI 状态：正常、空数据、错误、过期、加载。
- 回归风险：字段改名、数据源格式变化、构建产物泄露。

测试 ID 规则：

```text
<MODULE_PREFIX>-001
```

示例：

```text
TUD-001
SIG-001
WRITING-001
```

## 6. 实现前检查

实现前必须确认：

- 模块文档已经创建。
- MVP 范围和不做事项已经写清。
- 接口字段已经稳定到可以写样例数据。
- 测试用例已经覆盖隐私和错误处理。
- 总规划文档中的接续状态已指向该模块文档。

## 7. 完成后检查

模块完成后必须更新：

- 对应 `docs/feature-<module-slug>.md` 的当前状态。
- 总规划文档的接续状态卡。
- README 中公开项目说明，如果该模块已对外可见。
- 测试执行记录或手工验证记录。
